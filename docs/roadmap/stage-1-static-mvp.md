# Этап 1. Завершение статического MVP

Статус: согласован.

Короткое название: Готовый мир.

Краткое описание: Дополняем лор, расы и летописи, ускоряем статический фронтенд.

Родительский документ: [Продуктовый roadmap](../product-roadmap.md).

## Цель

Получить завершённую, стабильную и быстро загружающуюся статическую энциклопедию уже созданной части мира.

На этом этапе данные продолжают храниться во фронтенд-репозитории. Бэкенд, авторизация и пользовательское редактирование не добавляются.

## Результат для пользователя

Мастер и игроки могут открыть один публичный сайт и увидеть:

- интерактивную карту;
- книгу существующих героев;
- энциклопедию рас;
- две завершённые летописи;
- актуальный roadmap проекта.

Приложение быстрее загружается, не скачивает заранее ненужные арты, использует масштабируемую маршрутизацию и сохраняет единый визуальный язык.

## Порядок выполнения

### 1. Зафиксировать roadmap

- согласовать продуктовый документ;
- хранить общий roadmap и планы этапов в отдельных Markdown-файлах;

Результат:

- документация доступна в репозитории и готова стать источником данных для справочной страницы.

### 2. Заменить самописную маршрутизацию

До добавления новых страниц подключить React Router и удалить ручное сопоставление адреса с экранами.

На этапе 1 используется `createBrowserRouter` с обычными URL. Для GitHub Pages сборка публикует `404.html`, который загружает SPA на прямых внутренних адресах. React Router поддерживает декларативные route objects, параметры и ленивую загрузку реализаций маршрутов: [официальная документация](https://reactrouter.com/api/data-routers/createBrowserRouter).

Порядок работы:

1. Добавить актуальную стабильную и совместимую с текущими React, TypeScript и Vite версию `react-router` через npm.
2. Создать конфигурацию маршрутов в слое `app` и подключить `RouterProvider`.
3. Создать общий layout приложения с навигацией и `Outlet`.
4. Перенести существующие страницы на route objects.
5. Заменить ручные изменения адреса на `Link`, `NavLink` и router navigation.
6. Получать `region-id` через параметры маршрута, а не регулярное выражение.
7. Добавить отдельные состояния неизвестного маршрута и неизвестного региона.
8. Настроить lazy-loading страниц, чтобы новые разделы не попадали целиком в начальную загрузку.
9. После переключения удалить `useAppNavigation`, `regionHash.ts` и дублирующую логику.
10. Добавить SPA fallback `404.html` в GitHub Pages-сборку.
11. Проверить прямые ссылки, историю браузера и GitHub Pages-сборку.

Целевое дерево первого этапа:

```text
/
/heroes
/races
/region/:regionId
/roadmap
/*
```

Правила:

- приложение использует обычные browser URL без `#`;
- страницы остаются в FSD-слое `pages`;
- конфигурация маршрутов и layout находятся в `app`;
- пользовательская навигация оформляется в `features`;
- загрузка данных и бизнес-логика не переносятся в конфигурацию роутера;
- GitHub Pages и последующий облачный хостинг обязаны возвращать SPA entry point для внутренних маршрутов.

Результат:

- приложение использует единое декларативное дерево маршрутов;
- вложенные маршруты будущих редакторов можно добавлять без расширения самописного парсера;
- прямое открытие внутренних ссылок работает через SPA fallback;
- страницы могут загружаться отдельными чанками;
- неизвестный адрес показывает управляемую страницу ошибки.

### 3. Добавить страницу roadmap

- добавить справочный маршрут `/roadmap` после переключения на React Router;
- страница только отображает согласованный план и не позволяет его редактировать;
- сделать интерфейс лёгким и читаемым, без обязательной стилизации под D&D;
- добавить в основное меню отдельную вкладку roadmap;
- показывать общий горизонт и ссылки на подробности согласованных этапов.

Результат:

- пользователь может открыть актуальный roadmap внутри приложения;
- структура страницы соответствует зафиксированным Markdown-документам.

### 4. Добавить вторую завершённую кампанию

Автор принесёт сюжет и материалы из Figma и самостоятельно наполнит кампанию с помощью агента. Точная полнота данных определяется после изучения материалов.

Порядок работы:

1. Изучить сюжет и Figma-материалы.
2. Зафиксировать, какие сведения становятся каноном.
3. Определить доступную глубину структурированных данных.
4. Дополнить канонические файлы.
5. Подготовить публичный пересказ пройденной истории.
6. Связать кампанию с соответствующим регионом.
7. Добавить и зарегистрировать утверждённые арты.
8. Реализовать страницу завершённого региона.
9. Проверить страницу в браузере.

Минимальный результат:

- регион отмечен как завершённый;
- карта открывает страницу региона;
- страница содержит публичную летопись;
- отображаются утверждённые материалы из Figma;
- полные карточки героев на странице не дублируются.

Полный набор сцен, NPC, врагов и отдельный мастерский гайд заранее не обязателен. Он добавляется только при наличии достаточных исходных материалов.

### 5. Добавить энциклопедию рас

Страница рас на первом этапе является только справочной.

Порядок работы:

1. Дополнить канонический список рас.
2. Согласовать новые игровые особенности и формулировки.
3. При необходимости сначала описать новые поля в `content/README.md`.
4. Добавить данные в `content/races.json`.
5. Добавить утверждённые изображения рас.
6. При появлении новых ассетов обновить `assets/concepts/manifest.json`.
7. Реализовать маршрут `/races`.
8. Добавить пункт «Расы» в основное меню.
9. Проверить страницу в браузере.

На странице показываются:

- название;
- описание;
- игровая особенность;
- арт;
- связанные герои, если эта связь представлена в канонических данных.

На этапе 1 не добавляются:

- выбор расы;
- создание персонажа;
- изменение характеристик;
- пользовательское редактирование описаний.

### 6. Провести оптимизацию и техническую стабилизацию

Сначала измеряется исходное состояние. Числовые цели определяются после замеров, а не назначаются заранее.

#### Зафиксировать исходное состояние

- измерить размер production-сборки;
- зафиксировать объём и форматы изображений;
- проверить первоначальные сетевые запросы;
- определить ассеты, загружаемые раньше времени;
- зафиксировать заметные проблемы загрузки и отображения.

#### Оптимизировать загрузку

- подготовить облегчённые версии тяжёлых изображений;
- сохранять прозрачность там, где она нужна карте и персонажам;
- загружать арты только на использующих их маршрутах;
- применить отложенную загрузку подходящих изображений;
- проверить разделение кода по маршрутам;
- не ухудшить качество утверждённых визуальных материалов.

#### Усилить автоматические проверки

- проверять изменённые JSON через `jq empty`;
- проверять уникальность `id`;
- проверять ссылки между регионами, кампаниями, героями, NPC и врагами;
- запускать TypeScript-проверку и production-сборку;
- запускать сборку для GitHub Pages.

#### Сравнить результат

- повторить исходные измерения;
- сравнить загрузку до и после;
- зафиксировать достигнутое улучшение;
- перечислить оставшиеся тяжёлые ассеты и технические ограничения.

## Последовательность задач

```text
Документация roadmap
  ↓
Переход на React Router
  ↓
Страница roadmap
  ↓
Вторая завершённая кампания
  ↓
Энциклопедия рас
  ↓
Исходные замеры производительности
  ↓
Оптимизация и техническая стабилизация
  ↓
Финальная визуальная и техническая проверка
```

Вторая кампания и энциклопедия рас могут разрабатываться независимо, но оптимизацию следует проводить после добавления основных материалов этапа. Иначе замеры не будут отражать итоговый объём статического MVP.

## Критерии завершения

- доступны карта, герои, расы, roadmap и две завершённые летописи;
- страницы читают факты из канонических данных;
- приложение использует React Router и обычные URL без `#`;
- внутренние маршруты открываются напрямую через SPA fallback;
- кнопки назад и вперёд корректно переключают страницы;
- неизвестный маршрут и неизвестный регион обрабатываются явно;
- страницы разделяются на route-level чанки там, где это уменьшает начальную загрузку;
- начальная страница не загружает ненужные арты других разделов;
- карта сохраняет высоту окна и не создаёт скролл;
- нет ошибок в консоли и сломанных ссылок между сущностями;
- интерфейс проверен на ширинах 1024, 1280 px и мобильной ширине;
- проходят `npm run typecheck`, `npm run build` и `npm run build:pages`;
- результат оптимизации зафиксирован сравнением до и после;
- новые канонические факты перечислены в результате соответствующих контентных задач.

## Что не входит в этап

- серверное хранение данных;
- авторизация;
- создание и редактирование персонажей через интерфейс;
- сюжетный редактор;
- игровые комнаты;
- подключение игроков со своих устройств;
- изменение домашней d20-системы;
- восстановление отсутствующих деталей второй кампании без участия автора.

## Зависимости и открытые входные данные

- Для второй кампании нужны сюжет и материалы из Figma.
- Для расширения списка рас нужны названия, описания, механики и доступные арты.
- Точные цели производительности определяются после исходных измерений.

Эти входные данные уточняются внутри соответствующих задач и не блокируют утверждение общего плана этапа.
