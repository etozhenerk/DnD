# Этап 4. Создание и редактирование персонажей

Статус: согласован.

Родительский документ: [Продуктовый roadmap](../product-roadmap.md).

## Цель

Дать мастеру возможность создавать, проверять, публиковать и редактировать персонажей через сервис до появления Яндекс ID и пользовательских аккаунтов.

На этом этапе персонажи ещё не принадлежат пользователям и не добавляются в игровую партию. Мастер управляет общим каталогом канонических героев.

## Результат для пользователя

Мастер может:

- войти в закрытый раздел по простому паролю редактора;
- создать персонажа пошагово;
- сохранить незавершённый черновик;
- загрузить готовый портрет;
- увидеть книжный предпросмотр;
- получить ошибки и предупреждения проверки;
- опубликовать персонажа;
- подготовить новую ревизию существующего героя;
- восстановить предыдущую ревизию;
- убрать персонажа в архив без разрушения старых ссылок.

После публикации персонаж появляется в публичной книге героев через API.

## Согласованные решения

- До Яндекс ID используется один простой пароль редактора.
- Пароль проверяется бэкендом, а не фронтендом.
- После входа сервер создаёт защищённую сессию редактора.
- Используются состояния `draft`, `published` и `archived`.
- Изменение опубликованного персонажа выполняется через новую ревизию.
- Предыдущие ревизии сохраняются и могут быть восстановлены.
- В конструкторе выбираются только существующие опубликованные расы.
- Новая раса создаётся отдельной контентной задачей.
- Баланс проверяется мягко: предупреждения не блокируют публикацию мастером.
- Стандартные способности и предметы описываются машинно-читаемыми эффектами, совместимыми с локальным движком этапа 2.
- Уникальные сюжетные эффекты могут требовать ручного подтверждения мастера.
- Готовые портреты загружаются в Yandex Object Storage.
- AI-генерации внутри интерфейса нет.
- Исходные личные фотографии в сервис не загружаются.
- Физического удаления опубликованных персонажей нет.

## Пользовательские маршруты

Предлагаемые маршруты:

```text
/editor
/editor/characters
/editor/characters/new
/editor/characters/<character-id>
/editor/characters/<character-id>/preview
/editor/characters/<character-id>/history
```

Точные адреса могут быть скорректированы при проектировании навигации, но редактор остаётся отделённым от публичной книги героев.

## 1. Временный доступ редактора

### Сценарий входа

1. Мастер открывает редактор.
2. Фронтенд показывает нейтральную форму пароля.
3. Бэкенд проверяет пароль редактора.
4. После успешной проверки сервер создаёт ограниченную по времени сессию.
5. Браузер получает защищённую `HttpOnly` cookie.
6. Все операции записи требуют действующую сессию.

### Ограничения

- нет пользователей;
- нет ролей;
- нет восстановления пароля;
- нет интерфейса управления доступом;
- нет нескольких паролей;
- механизм полностью заменяется Яндекс ID на этапе 5.

Пароль хранится только в серверной конфигурации Yandex Cloud или Lockbox и не попадает во фронтенд или репозиторий.

## 2. Каталог персонажей мастера

Редактор показывает общий список персонажей.

Для каждой записи отображаются:

- имя;
- портрет или заглушка;
- раса;
- игровая роль;
- статус;
- дата последнего изменения;
- наличие неопубликованного черновика;
- текущая опубликованная ревизия.

Доступные действия:

- создать персонажа;
- продолжить черновик;
- редактировать опубликованного персонажа;
- открыть предпросмотр;
- посмотреть историю ревизий;
- опубликовать;
- архивировать;
- вернуть из архива.

## 3. Пошаговое создание персонажа

### Шаг 1. Основное

- имя;
- стабильный `id`;
- местоимения;
- существующая опубликованная раса;
- игровая роль;
- краткая фраза-концепт.

`id` создаётся в lower-kebab-case, проверяется на уникальность и не меняется после первой публикации.

### Шаг 2. История и характер

- история;
- мотивация;
- характер;
- сильная сторона;
- слабость;
- личный юмор;
- отношения с существующими персонажами, местами или конфликтами.

Художественный текст хранится отдельно от формальных игровых эффектов.

### Шаг 3. Характеристики

- текущий и максимальный HP для канонической карточки;
- AC;
- сила;
- ловкость;
- телосложение;
- мудрость;
- интеллект;
- харизма.

Состояние персонажа в конкретной игровой сессии на этом этапе не создаётся.

### Шаг 4. Способности

Для каждой способности:

- стабильный `id` внутри персонажа;
- название;
- художественное описание;
- триггер;
- проверка при наличии;
- формальный эффект;
- уровень автоматизации: автоматический или с подтверждением мастера;
- тип действия и допустимые цели;
- формула броска и тип защиты: AC, DC или решение мастера;
- стандартные эффекты: урон, лечение, состояние, изменение ресурса;
- длительность эффекта при наличии;
- `uses.scope` и `uses.max` для ограниченного использования.

Допустимые области восстановления берутся из действующих правил.

### Шаг 5. Предметы

Для каждого предмета:

- `id`;
- название;
- описание;
- формальный эффект;
- машинно-читаемый стандартный эффект при наличии;
- допустимую цель и способ активации;
- уровень автоматизации;
- заряды при наличии.

Художественное описание не заменяет формальную механику. Если эффект невозможно безопасно автоматизировать, он явно помечается как требующий решения мастера.

### Шаг 6. Внешность

- готовый портрет;
- доступный alt-текст;
- краткое описание внешности;
- ссылки на визуальные концепты расы при необходимости.

### Шаг 7. Проверка и публикация

- сводка ошибок;
- предупреждения баланса;
- книжный предпросмотр;
- подтверждение канонизации;
- публикация.

Черновик автоматически сохраняется между шагами.

## 4. Черновики и ревизии

### Новый персонаж

- создаётся как `draft`;
- не виден в публичной книге;
- может быть незавершённым;
- становится каноническим только после публикации.

### Редактирование опубликованного персонажа

- публичная ревизия остаётся неизменной;
- мастер получает отдельный черновик следующей ревизии;
- изменения можно предварительно просмотреть;
- публикация атомарно заменяет публичную ревизию;
- старая ревизия остаётся в истории.

### Восстановление

- мастер выбирает предыдущую ревизию;
- система создаёт на её основе новый черновик;
- восстановление также проходит проверку и публикацию;
- история не переписывается задним числом.

### Архив

- архивный персонаж скрывается из обычной книги;
- старые ссылки и летописи продолжают разрешаться;
- архивирование обратимо;
- физического удаления опубликованной сущности нет.

## 5. Портреты и Object Storage

Для пользовательских загрузок используется Yandex Object Storage, совместимый с S3 API: [официальная документация](https://yandex.cloud/en/docs/storage/).

### Поток загрузки

```text
Выбор готового портрета
  ↓
Запрос временной ссылки у бэкенда
  ↓
Загрузка в закрытое хранилище черновиков
  ↓
Серверная проверка
  ↓
Предпросмотр
  ↓
Публикация оптимизированной версии
```

Yandex Object Storage поддерживает pre-signed URL для ограниченной по времени загрузки в закрытый bucket: [документация](https://yandex.cloud/en/docs/storage/operations/objects/link-for-upload).

### Проверки файла

- допустимый формат JPEG, PNG или WebP;
- допустимый размер файла;
- минимальные и максимальные размеры изображения;
- соответствие заявленного MIME фактическому содержимому;
- отсутствие исполняемого содержимого;
- удаление лишних метаданных при подготовке производной версии;
- уникальное имя объекта;
- отсутствие перезаписи старого портрета.

### Правила приватности

- загружается только готовый производный портрет;
- исходная личная фотография не загружается;
- исходная личная фотография не коммитится;
- черновой файл не становится публичным до публикации персонажа;
- отклонённые и осиротевшие черновые загрузки очищаются по отдельному техническому правилу.

AI-генерация и обработка портрета внутри пользовательского сервиса не входят в этап.

## 6. Проверки персонажа

### Блокирующие ошибки

- отсутствуют обязательные поля;
- `id` невалиден или занят;
- выбрана неизвестная или неопубликованная раса;
- числовые поля имеют неверный формат;
- ограниченная способность не содержит `uses.scope` или `uses.max`;
- автоматический эффект не содержит тип цели, формулу или исполняемое действие;
- формула использует неизвестную характеристику, состояние или область восстановления;
- ссылка отношения ведёт на несуществующую сущность;
- портрет имеет недопустимый тип;
- данные не соответствуют API-контракту.

### Мягкие предупреждения

- HP или AC значительно отличаются от текущих ориентиров;
- персонаж одновременно имеет слишком много сильных ролей;
- способность заметно сильнее существующих аналогов;
- эффект помечен автоматическим, хотя содержит неоднозначное сюжетное решение;
- отсутствует слабость или сюжетная связь;
- слишком много способностей или предметов;
- книжный разворот перегружен текстом.

Мастер может опубликовать персонажа с предупреждениями. Предупреждения сохраняются в отчёте публикации, но не меняют данные автоматически.

## 7. Предпросмотр

Предпросмотр использует тот же UI-компонент, что и публичная книга, но получает данные черновика.

Необходимо проверить:

- все вкладки персонажа;
- длинное имя;
- длинные названия способностей;
- переполнение текста;
- одинаковую высоту книжного разворота;
- отсутствие внутреннего вертикального скролла статистики и навыков;
- корректное отображение автоматических и подтверждаемых мастером эффектов;
- стрелки перелистывания;
- alt-текст портрета;
- ширины 1024, 1280 px и мобильную компоновку.

## 8. Предлагаемая модель данных

```text
characters
character_revisions
character_drafts
character_assets
editor_sessions
```

### `characters`

Стабильная идентичность персонажа, статус и ссылка на текущую опубликованную ревизию.

### `character_revisions`

Неизменяемые опубликованные версии персонажа.

### `character_drafts`

Изменяемые черновики нового персонажа или следующей ревизии.

### `character_assets`

Связи персонажа, ревизии и объектов Object Storage.

### `editor_sessions`

Временные серверные сессии редактора по общему паролю.

Пользователь и владелец персонажа в модель этапа 4 не входят.

## 9. API

Публичное чтение остаётся доступным без пароля.

Операции редактора требуют действующую серверную сессию:

```text
POST   /api/v1/editor/session
DELETE /api/v1/editor/session
GET    /api/v1/editor/characters
POST   /api/v1/editor/characters
GET    /api/v1/editor/characters/{id}/draft
PATCH  /api/v1/editor/characters/{id}/draft
POST   /api/v1/editor/characters/{id}/publish
POST   /api/v1/editor/characters/{id}/archive
POST   /api/v1/editor/characters/{id}/restore
GET    /api/v1/editor/characters/{id}/revisions
POST   /api/v1/editor/uploads
```

Точный контракт фиксируется в OpenAPI до реализации фронтенда.

6. Подключить Object Storage и загрузку портретов.

12. Провести серверные, интеграционные и визуальные проверки.
13. Проверить обновление публичной книги после публикации.

- создание и автосохранение черновика;

- мягкие предупреждения;
- атомарная публикация;

### Фронтенд

## Критерии завершения

- мастер входит в редактор по серверному паролю;
- без сессии нельзя изменить данные;
- можно создать полный черновик персонажа;
- черновик сохраняется между открытиями;
- можно выбрать только опубликованную расу;
- проверки домашней системы работают;
- стандартные способности и предметы публикуются с валидными машинно-читаемыми эффектами;
- неоднозначные способности могут быть опубликованы как требующие подтверждения мастера;
- предупреждения баланса не блокируют мастера;
- готовый портрет загружается через Object Storage;
- личная исходная фотография не требуется и не сохраняется;
- предпросмотр соответствует публичной книге;
- публикация создаёт неизменяемую ревизию;
- персонаж появляется в публичной книге;
- опубликованные изменения не видны до новой публикации;
- предыдущую ревизию можно восстановить через новый черновик;
- архивирование не ломает старые ссылки;
- серверные, интеграционные и визуальные тесты проходят;
- OpenAPI и документация обновлены.

## Что не входит в этап

- Яндекс ID;
- пользователи и роли;
- самостоятельное редактирование игроками;
- привязка персонажа к владельцу;
- создание новых рас;
- AI-генерация персонажа;
- AI-генерация портрета;
- загрузка исходных личных фотографий;
- выбор персонажа для партии;
- добавление персонажа в кампанию;
- текущее HP и эффекты игровой сессии;
- физическое удаление опубликованного персонажа;
- редактор кампаний.

## Открытые технические решения внутри этапа

- время жизни сессии редактора;
- конкретный способ хеширования пароля;
- ограничения размера и разрешения портрета;
- формат оптимизированной производной версии;
- один bucket с префиксами или отдельные buckets для черновиков и публикаций;
- срок очистки незавершённых загрузок;
- точный формат хранения ревизии;
- стратегия конкурентного редактирования.

Решения, меняющие пользовательский сценарий, публичность данных или правила канонизации, согласуются отдельно.
