# Инструкция для агентов

## Цель проекта

Развивать «Хроники Восьми Земель» — кликабельный React-сервис авторских D&D-кампаний для одного общего экрана. Мастер управляет страницей и рассказывает историю игрокам.

Текущий scope MVP:

- без авторизации и игровых комнат;
- без отдельного интерфейса мастера;
- карта, книга героев и публичные летописи регионов;
- домашняя облегчённая d20-система;
- статический фронтенд, публикуемый через GitHub Pages.

## Начало работы

Перед изменениями:

1. Полностью прочитать этот файл.
2. Проверить `git status --short` и не перезаписывать несвязанные изменения пользователя.
3. Для контентной задачи прочитать `content/README.md`, `content/rules.json` и затронутые файлы кампании.
4. Для фронтенда определить FSD-слой и проверить существующие компоненты перед созданием нового.
5. Использовать только npm. Не добавлять pnpm, yarn или альтернативные lock-файлы.

## Команды проекта

```bash
npm ci
npm run dev
npm run typecheck
npm run build
npm run preview
npm run build:pages
```

`package-lock.json` обязателен. `node_modules/` и `dist/` не коммитятся.

## Источники истины

Приоритет данных:

1. `content/**/*.json` — канонические структурированные данные приложения.
2. `content/campaigns/*-guide.md` — человекочитаемые сценарии мастера; они должны соответствовать JSON.
3. `content/README.md` — схема сущностей и термины.
4. `assets/concepts/manifest.json` и `assets/concepts/` — утверждённые визуальные сущности и референсы.
5. `README.md` — актуальное пользовательское описание проекта, запуска и публикации.

Игнорируемые исходные материалы не являются каноном, не должны упоминаться как рабочий вход в README и не могут быть обязательной зависимостью скиллов или приложения.

При конфликте старого исходника и согласованного `content/` не исправлять канон молча. Описать конфликт и запросить решение пользователя.

## Архитектура фронтенда

Стек: React 19, TypeScript, Vite, CSS Modules, FSD.

```text
src/
  app/         композиция приложения
  pages/       страницы карты, героев и региона
  widgets/     крупные самостоятельные блоки
  features/    пользовательские действия и навигация
  entities/    модели и UI игровых сущностей
  shared/      конфигурация, утилиты, ассеты и глобальные стили
```

Правила FSD:

- верхний слой может импортировать нижний, но не наоборот;
- `shared` не импортирует `entities`, `features`, `widgets` или `pages`;
- `entities` не импортируют `features`, `widgets` или `pages`;
- не складывать бизнес-логику в `app` и `pages`;
- перед добавлением нового слоя или сущности проверять существующую структуру.

## Правила компонентов и стилей

- Один React-компонент — один `.tsx`-файл.
- Стили компонента хранятся рядом в `Component.module.css`.
- Не использовать глобальные классы для локального UI.
- Все цвета, альфа-значения, градиенты и тени задавать через переменные `src/shared/styles/variables.css`.
- В `*.module.css` запрещены прямые `#hex`, `rgb()`, `rgba()`, `hsl()` и `hsla()`.
- Преобразования, генераторы, парсеры и повторно используемые функции выносить в `shared/lib` или `model` соответствующего слоя.
- Не добавлять UI-библиотеку или пакет иконок, если задачу можно решить текущими средствами и ассетами.
- Сохранять доступные подписи, клавиатурное управление и `focus-visible`.

## Навигация и утверждённый UX

Маршруты реализуются через React Router и обычные URL:

- `/` — карта;
- `/heroes` — книга героев;
- `/region/<region-id>` — летопись региона.
- `/roadmap` — продуктовый план развития проекта.

Инварианты интерфейса:

- верхнее меню содержит отдельные пункты «Карта» и «Герои»;
- карта занимает ровно высоту окна браузера и не создаёт вертикальный или горизонтальный скролл;
- регионы карты кликабельны через SVG-полигоны;
- герои находятся на отдельной странице, а не под картой;
- высота книжного разворота героя не меняется при перелистывании;
- стрелки перелистывания расположены у левого и правого края книги;
- статистика и навыки помещаются в разворот без внутреннего вертикального скролла;
- страница завершённого Нор’Иль’Скальда содержит пересказ сюжета, но не дублирует карточки героев;
- страницы карты, героев и регионов используют единый тёмный сказочный визуальный язык;
- пергаментная книга остаётся контрастным игровым объектом на тёмном фоне.

Не возвращать удалённые паттерны: правую легенду регионов на карте, героев под картой, стеклянные dashboard-карточки и белый фон всей страницы.

## Работа с данными во фронтенде

- Интерфейс читает лор и характеристики из `content/`, а не хранит их копии внутри компонентов.
- Новый факт сначала добавляется в канонический JSON, затем отображается во фронтенде.
- Новое поле сначала описывается в `content/README.md`, затем добавляется в данные, TypeScript-модель и UI.
- Не показывать игрокам секрет мастера без отдельного публичного поля или режима.
- Пути к изображениям разрешать через существующий asset resolver.
- Не переименовывать стабильные `id` без миграции всех ссылок и маршрутов.

## Обязательная маршрутизация по локальным скиллам

- Новая область или полный цикл кампании: `skills/create-dnd-campaign/SKILL.md` как оркестратор и `docs/campaign-skill-workflow.md` как контракт передачи.
- Концепция: `define-dnd-campaign-concept`; сюжетный каркас: `outline-dnd-campaign`; граф: `design-dnd-story-graph`.
- Обычный сюжетный NPC: `create-dnd-npc`; игровой герой: `create-dnd-character`; враг, злодей или боевая группа: `create-dnd-enemy`.
- Полный текст сцен: `write-dnd-scene-script`; диалоговые пресеты мастера: `write-dnd-dialogue`; формальные механики: `design-dnd-gameplay`.
- Редакторская приёмка до артов: `review-dnd-campaign`.
- Арты: `plan-dnd-campaign-art` → `generate-dnd-campaign-art` → `review-dnd-campaign-art` → `publish-dnd-campaign-assets`.
- Реализация: `plan-dnd-campaign-implementation` → `publish-dnd-campaign` → `implement-dnd-campaign-ui`.
- Локальная игра: `model-dnd-game-session` → `implement-dnd-rules-engine` → `implement-dnd-npc-behavior` → `implement-dnd-dialogue-console` → `implement-dnd-gm-console`.
- Проверка игровых сценариев: `test-dnd-gameplay-scenarios`; финальная приёмка: `verify-dnd-campaign-release`.
- Структура данных, связи, правила или несколько типов сущностей: `maintain-dnd-world`; применять его последним для проверки целостности.

Каждый этап изменяет только свой рабочий артефакт. Следующий этап запускается после явного утверждения пользователя; изменение раннего этапа помечает зависимые результаты устаревшими.

Всегда сообщать пользователю, какой локальный скилл применяется и почему.

## Маршрутизация frontend и design-задач

- Реализация экрана по Figma: `$figma` и `$figma-implement-design`.
- Создание или обновление полного экрана в Figma: `$figma-use` и `$figma-generate-design`.
- Компоненты и дизайн-система в Figma: `$figma-use`, `$figma-generate-library` и при необходимости `$figma-create-design-system-rules`.
- Связь Figma-компонентов с React-компонентами: `$figma-code-connect-components`.
- Визуальная и интерактивная проверка frontend-изменений: встроенный браузер Codex через `browser:control-in-app-browser` и локальный `verify-dnd-campaign-release`.
- Standalone `$playwright`, `$playwright-interactive` и `playwright-cli` не запускать автоматически; использовать их только после отдельного разрешения пользователя, если встроенный браузер недоступен.
- Контрольные скриншоты делать средствами встроенного браузера; `$screenshot` использовать только когда явно нужен снимок окна или всего рабочего стола.
- Для существенной переработки UI сначала зафиксировать контрольные viewport и референсы, затем проверить итоговый рендер скриншотом на тех же размерах. Успешные `typecheck` и `build` не заменяют визуальную проверку.

Всегда сообщать пользователю, какие frontend/design-скиллы применяются и для какой части работы.

## Правила контента

- Сохранять домашнюю систему из `content/rules.json`; не заменять её D&D 5e.
- Сохранять русский язык, личный юмор, абсурд и местоимения персонажей.
- Использовать lower-kebab-case для `id`.
- Разделять художественный текст мастера и формальный игровой эффект.
- Для ограниченного умения указывать `uses.scope` и `uses.max`.
- Допустимые области восстановления: `turn`, `round`, `battle`, `location`, `campaign`.
- Новый канонический факт явно перечислять в результате работы.
- Не считать новый портрет каноническим до одобрения пользователя.

Если герой основан на реальном человеке:

- для знакомого или другого частного человека фотография обязательна как визуальный референс;
- для знаменитого человека достаточно точного имени и выбранного публичного образа; фото или ссылка нужны для конкретного периода или внешности;
- характер брать из описания пользователя или явно выбранного публичного образа, но не выводить из фотографии;
- сохранять узнаваемость;
- учитывать стиль расы и существующей команды;
- согласовывать степень расовой трансформации;
- не коммитить исходную личную фотографию, имя частного источника и путь к фото.

## Визуальные ассеты

- Использовать утверждённые файлы из `assets/concepts/`.
- Новые производные концепты регистрировать в `assets/concepts/manifest.json`.
- Не коммитить личные фотографии, тяжёлые исходники и временные рендеры.
- Для карты регионов сохранять прозрачный PNG и аккуратный контур суши; облачный остров включает облако и относящуюся к нему воду.
- При работе по Figma сначала сверяться с референсами из `assets/concepts/style/`.

## Проверка контентных изменений

1. Запустить `jq empty` для изменённых JSON.
2. Проверить уникальность `id`.
3. Проверить ссылки на героев, NPC, врагов, встречи и локации.
4. Сверить JSON кампании и соответствующий `*-guide.md`.
5. Проверить баланс для пяти героев либо явно указать другой размер группы.

## Проверка фронтенда

Минимум:

```bash
npm run typecheck
npm run build
```

Перед публикацией также:

```bash
npm run build:pages
```

Для изменённого UI проверить в браузере:

- отсутствие ошибок в консоли;
- `document.documentElement.scrollWidth === document.documentElement.clientWidth`;
- карту на высоте окна браузера;
- маршруты карты, героев, региона и roadmap;
- клики по SVG-регионам;
- перелистывание каждого затронутого героя;
- одинаковую высоту книги до и после перелистывания;
- отсутствие внутреннего скролла у статистики и навыков;
- длинные имена и заголовки на ширинах 1024 и 1280 px;
- мобильную компоновку, если менялись адаптивные стили.

Для кампании с локальным игровым циклом дополнительно проверить применение навыка и расход использований, инициативу и раунды, решение NPC и override мастера, выбор диалогового пресета, ручное изменение HP/AC/атаки/инвентаря, отмену, восстановление из `localStorage` и влияние раннего решения на позднюю сцену или финал.

## GitHub Pages

- Production workflow: `.github/workflows/deploy-pages.yml`.
- Pages-сборка: `npm run build:pages`.
- Репозиторий: `etozhenerk/DnD`.
- Публичный base path: `/DnD/` в режиме `github-pages`.
- URL: `https://etozhenerk.github.io/DnD/`.
- Обычные browser-маршруты должны иметь SPA fallback на `index.html`; для GitHub Pages поддерживать `404.html`, после переезда — fallback в API Gateway.
- При переименовании репозитория синхронно обновить `base` и README.

## Git и безопасность

- Не коммитить `.env`, зависимости, сборки, приватные ключи и личные фотографии.
- Не менять и не удалять несвязанные пользовательские правки.
- Не использовать destructive-команды без явного запроса.
- Не выполнять commit, push или deployment без явного разрешения пользователя.
- Перед handoff показать изменённые файлы, проверки, допущения и открытые вопросы.
