# Конвейер создания кампании с локальными скиллами

Этот документ задаёт общий контракт между скиллами этапа 2. Он описывает рабочие черновики, а не канон мира. Каноном остаются только опубликованные файлы из `content/` и утверждённые записи `assets/concepts/manifest.json`.

## Рабочая папка кампании

До публикации материалы хранить в `docs/campaigns/<campaign-id>/`:

```text
workflow.json
concept.md
outline.md
story-graph.json
story-graph.md
cast.md
script.md
dialogue.json
dialogue.md
gameplay.json
gameplay-notes.md
review.md
art-plan.json
implementation-plan.md
retrospective.md
```

`<campaign-id>` во всех реальных путях заменять на lower-kebab-case идентификатор кампании.

## Состояния этапов

`workflow.json` хранит для каждого артефакта одно состояние:

- `draft` — подготовлен, но не утверждён;
- `approved` — пользователь явно утвердил результат;
- `stale` — входной материал изменился после утверждения;
- `published` — результат перенесён в канон или приложение.

Скилл не может сам поставить своему результату `approved`. Автономно подготовленный результат остаётся `draft` до явного подтверждения пользователя.

## Правила передачи

1. Читать `AGENTS.md`, этот документ, `workflow.json` и только необходимые утверждённые входы.
2. Не начинать этап, если обязательный вход отсутствует, имеет состояние `draft` или `stale`.
3. Изменять только артефакты своего этапа и их запись в `workflow.json`.
4. Не переносить факты в `content/`, ассеты в manifest или код в `src/` до специализированного этапа публикации.
5. Показывать пользователю результат, допущения, открытые решения и последствия утверждения.
6. Задавать не более трёх вопросов за один раунд и сначала искать ответы в каноне.

## Инвалидация

Изменение раннего этапа переводит зависимые результаты в `stale`:

```text
concept -> outline -> story-graph -> cast -> script
        -> dialogue + gameplay -> review -> art-plan
        -> implementation-plan -> published content, assets and code
```

Если изменение локально и не влияет на зависимый артефакт, скилл обязан объяснить это. Молчаливо считать зависимый результат актуальным нельзя.

## Контрольные точки

Отдельное подтверждение пользователя требуется после концепции, каркаса, графа, состава персонажей, сценария и реплик, механик, редакторской проверки, арт-плана, каждого финального арта, плана реализации, публикации данных и реализации игрового интерфейса.

## Разделение данных

- Художественный текст хранить отдельно от исполняемых эффектов.
- Публичное содержание сцены хранить отдельно от секретов мастера.
- Канонические характеристики не менять во время прохождения: игра использует сессионный снимок.
- Ручные изменения мастера, автоматические эффекты и решения NPC записывать как события.
- Диалоговые пресеты выбирает мастер; поведение NPC в бою выбирает действие и цель по формальным приоритетам.

## Персонаж на основе реального человека

- Для знакомого или другого частного человека принимать описание характера и фотографию; фото использовать только для внешнего сходства.
- Для знаменитого человека принимать точное имя и выбранный публичный образ; конкретный визуальный референс запрашивать только при необходимости зафиксировать период или внешность.
- Не выводить характер, здоровье, происхождение и другие свойства из изображения.
- Отделять факты и черты, данные пользователем, от творческой интерпретации персонажа.
- Отдельно утверждать паспорт характера, уровень сходства, расовую трансформацию и финальный арт.
- Не записывать личное фото, имя частного источника и путь к нему в проект. Канонические данные описывают вымышленного персонажа, а не источник вдохновения.

## Виртуальный прогон без артефактов

Этот режим включается только по явному запросу пользователя провести кампанию виртуально, без файлов или без артефактов.

1. До начала зафиксировать `git status --short`; после завершения повторить проверку и подтвердить отсутствие изменений.
2. Не создавать, не редактировать, не удалять и не перемещать файлы в `docs/`, `content/`, `assets/`, `src/` и других каталогах проекта.
3. Возвращать компактные гипотетические артефакты прямо в ответе. Все они имеют смысл состояния `draft` и не становятся каноном.
4. Если пользователь явно передал полную автономию, условно проходить контрольные точки только для проверки зависимостей конвейера. Не присваивать результатам реальные `approved` или `published`.
5. Не вызывать генерацию изображений, публикацию ассетов, запись кода или браузерную проверку. Соответствующие скиллы использовать только как контракты для арт-плана, плана реализации и критериев приёмки.
6. Команду считать выполненной только после фактического запуска и получения её вывода. Сюжетный граф проверять через stdin командой `node skills/design-dnd-story-graph/scripts/validate-story-graph.mjs -`; если это невозможно, написать «проверено рассуждением, валидатор не запускался».
7. В финале разделить фактически выполненные проверки и гипотетические результаты прогона.

## Финальная проверка

После публикации применять `$maintain-dnd-world`, затем `$verify-dnd-campaign-release`. Визуальную и интерактивную проверку выполнять во встроенном браузере Codex. Не запускать standalone Playwright без отдельного разрешения пользователя.
