# Этап 3. Бэкенд-сервис и перенос данных

Статус: согласован.

Короткое название: Бэкенд и база.

Краткое описание: Переносим данные в Go-сервис и PostgreSQL, а фронтенд — в Yandex Cloud.

Родительский документ: [Продуктовый roadmap](../product-roadmap.md).

## Цель

Создать самостоятельный Go-сервис, перенести все текущие структурированные данные в PostgreSQL, полностью проверить миграцию и API, переключить React-фронтенд с локальных JSON на сервер и перенести production-фронтенд с GitHub Pages в Yandex Cloud.

Этап также переводит репозиторий к полноценной структуре монорепы с независимыми приложениями фронтенда и бэкенда и отдельным слоем концептов.

## Результат для пользователя

После переключения внешний вид и возможности существующего приложения не должны ухудшиться:

- карта продолжает открывать регионы;
- книга героев показывает тех же персонажей;
- энциклопедия рас сохраняет все данные;
- завершённые летописи доступны по прежним маршрутам;
- фронтенд получает структурированную информацию через API;
- база данных становится рабочим источником истины;
- production-фронтенд и API доступны через единый HTTPS-адрес Yandex Cloud.

Пользовательское создание и редактирование данных появится на следующих этапах.

## Согласованные решения

- Репозиторий становится монорепой.
- Фронтенд и бэкенд находятся в отдельных корневых каталогах.
- Концепты находятся на одном уровне с приложениями.
- Бэкенд пишется на Go.
- Используется PostgreSQL.
- API использует REST, JSON и документируется через OpenAPI.
- В базу переносятся все текущие структурированные данные.
- Сначала сервис тестируется независимо, затем переключается фронтенд.
- PostgreSQL становится рабочим источником истины.
- Исходные JSON сохраняются как неизменяемый исторический снимок в `concepts/`.
- JSON из `concepts/` не используется приложениями во время работы.
- До переключения GitHub Pages используется как временная production-площадка.
- Этап 3 завершается полным переносом production-фронтенда и бэкенда в Yandex Cloud.
- Один API Gateway раздаёт React-приложение из Object Storage и направляет `/api/*` в Go-сервис.
- Фронтенд и API работают с одного origin.
- После переключения GitHub Pages больше не считается production-площадкой.
- Собственный домен не блокирует этап: сначала допустим технический HTTPS-адрес API Gateway.
- Основной вариант прототипа: Serverless Containers, Container Registry, Managed PostgreSQL и Lockbox.
- Node.js для бэкенда не рассматривается.
- Резервное копирование не входит в scope этапа.

## Целевая структура монорепы

```text
dnd/
├── frontend/
│   ├── src/
│   ├── public/
│   ├── package.json
│   ├── package-lock.json
│   └── vite.config.ts
│
├── backend/
│   ├── cmd/
│   │   ├── api/
│   │   └── importer/
│   ├── internal/
│   ├── migrations/
│   ├── openapi/
│   ├── go.mod
│   ├── go.sum
│   └── Dockerfile
│
├── concepts/
│   ├── content-snapshot/
│   │   ├── rules.json
│   │   ├── races.json
│   │   ├── characters.json
│   │   ├── world-map.json
│   │   └── campaigns/
│   ├── assets/
│   └── manifest.json
│
├── docs/
├── skills/
├── .github/workflows/
├── AGENTS.md
└── README.md
```

### `frontend/`

Содержит только React-приложение, его конфигурацию, тесты и необходимые runtime-ассеты.

Правило «использовать только npm» продолжает действовать для фронтенда. После реструктуризации оно должно быть уточнено в `AGENTS.md`, чтобы не запрещать стандартные Go-инструменты внутри `backend/`.

### `backend/`

Содержит Go-сервис, API, доменную логику, работу с PostgreSQL, миграции и одноразовый импорт исходных данных.

Бэкенд является одним модульным приложением, а не набором микросервисов.

### `concepts/`

Содержит исходные материалы и исторические снимки, но не участвует в runtime приложений.

- `content-snapshot/` хранит финальное состояние JSON перед миграцией.
- `assets/` хранит утверждённые визуальные концепты.
- `manifest.json` описывает утверждённые визуальные материалы.

После миграции JSON не обновляются параллельно с базой и не считаются рабочим источником истины.

### `docs/` и `skills/`

- `docs/` содержит продуктовые и технические решения.
- `skills/` содержит локальные скиллы агентов и обновляется вместе с изменением источников истины и структуры проекта.

## Технологический подход

### Go-сервис

Рекомендуемый минимальный набор:

- стандартная библиотека Go;
- `chi` для HTTP-маршрутизации;
- `pgx` для подключения к PostgreSQL;
- `sqlc` для типобезопасной генерации кода из SQL;
- SQL-миграции;
- OpenAPI 3 для контракта;
- Docker для одинакового запуска локально и в облаке.

Точные версии и инструмент запуска миграций выбираются во время технического проектирования и фиксируются в lock-файлах или эквивалентных механизмах Go.

### Один модульный сервис

Внутренние модули соответствуют предметным областям:

```text
rules
races
characters
world
campaigns
gameplay
inventory
health
```

Модули разделяют модели и код, но разворачиваются одним приложением и используют одну базу данных.

### REST и OpenAPI

API получает версию в пути, например `/api/v1`. Контракт описывает:

- публичные модели;
- обязательные поля;
- форматы идентификаторов;
- ответы об ошибках;
- пагинацию там, где она понадобится;
- health endpoints;
- разделение публичных и мастерских данных.

Примеры ресурсов:

```text
GET /api/v1/rules
GET /api/v1/races
GET /api/v1/characters
GET /api/v1/characters/{id}
GET /api/v1/world-map
GET /api/v1/regions/{id}
GET /api/v1/campaigns/{id}/chronicle
GET /health/live
GET /health/ready
```

Точный набор endpoints определяется после проектирования публичных сценариев фронтенда.

## Модель хранения

Используется гибридная реляционная модель:

- самостоятельные сущности получают отдельные таблицы;
- стабильные `id` сохраняются;
- связи защищаются внешними ключами;
- повторяемые связи хранятся в таблицах отношений;
- сложные вложенные эффекты без самостоятельного жизненного цикла могут храниться в JSONB;
- художественный текст отделяется от формальных механик;
- публичные сведения отделяются от секретов мастера.

Основные группы данных:

- правила и состояния;
- расы и расовые особенности;
- персонажи, характеристики, способности и предметы;
- машинно-читаемые игровые эффекты;
- общий инвентарь и текущее количество постоянных предметов;
- карта, регионы и полигоны;
- кампании;
- локации и сцены;
- NPC и враги;
- встречи;
- условия, флаги и счётчики;
- публичные летописи.

Точная схема документируется отдельным техническим решением до создания миграций.

## Публичные и мастерские данные

В базу переносятся все данные, но публичный API не должен автоматически отдавать секреты мастера.

- Публичные endpoints возвращают только данные, необходимые текущему фронтенду.
- Мастерские поля остаются в базе без публичного endpoint до появления серверной авторизации.
- OpenAPI отдельно описывает публичные DTO, а не повторяет внутреннюю модель базы.
- Тесты проверяют отсутствие мастерских полей в публичных ответах.

Клиентский спойлер-барьер этапа 2 не заменяет это разделение.

## Порядок выполнения

### 1. Зафиксировать состояние перед миграцией

Этап начинается после завершения этапа 2.

- убедиться, что финальные JSON валидны;
- проверить уникальность `id`;
- проверить все ссылки;
- зафиксировать количество сущностей каждого типа;
- переместить JSON в `concepts/content-snapshot/`;
- отметить снимок как неизменяемый;
- зафиксировать версию снимка в документации миграции.

### 2. Перестроить репозиторий без изменения поведения

Сначала выполняется только архитектурное перемещение:

- перенести React-приложение в `frontend/`;
- перенести утверждённые визуальные концепты в `concepts/assets/`;
- обновить пути, конфигурацию Vite и GitHub Pages;
- обновить `AGENTS.md`, README и локальные скиллы;
- сохранить текущие маршруты и внешний вид;
- запустить frontend-проверки и визуальный smoke test.

После этого создаётся каталог `backend/`.

### 3. Спроектировать базу и API

- сопоставить каждый JSON-объект таблицам и полям;
- определить реляционные связи;
- выбрать допустимые JSONB-поля;
- описать публичные DTO;
- описать OpenAPI-контракт;
- проверить, что схема поддерживает текущие сущности без потери данных;
- провести отдельное согласование любых изменений формы данных.

### 4. Создать каркас Go-сервиса

- конфигурация;
- HTTP-сервер;
- подключение PostgreSQL;
- обработка ошибок;
- структурированные логи;
- корректное завершение процесса;
- health endpoints;
- миграции;
- локальный Docker-запуск;
- автоматические тесты.

### 5. Создать импорт исходных данных

Отдельная команда `importer` читает зафиксированный JSON-снимок и заполняет чистую базу.

Импорт должен:

- выполняться транзакционно;
- не оставлять частично заполненную базу;
- сохранять стабильные идентификаторы;
- проверять ссылки;
- отклонять неизвестные или некорректные поля;
- выдавать отчёт о количестве импортированных сущностей;
- поддерживать повторяемый запуск на пустой базе;
- завершаться ошибкой при расхождении инвариантов.

Импортер не является постоянным каналом редактирования данных после переключения.

### 6. Реализовать read-only API

- реализовать публичные endpoints;
- не добавлять пользовательское редактирование;
- не публиковать мастерские данные;
- настроить локальную разработку фронтенда с API;
- не требовать CORS для production, где фронтенд и `/api/*` работают с одного origin;
- добавить кэширование или ETag для редко меняющегося публичного контента;
- проверить ответы по OpenAPI.

### 7. Проверить равенство данных

До переключения фронтенда автоматически сравниваются:

- количество сущностей каждого типа;
- стабильные `id`;
- правила и состояния;
- расы и особенности;
- герои, характеристики, способности и предметы;
- игровые эффекты и статический общий инвентарь;
- регионы и полигоны;
- кампании, локации и сцены;
- NPC, враги и встречи;
- связи между сущностями;
- публичные летописи;
- отсутствие секретов мастера в публичном API.

Локальное незавершённое прохождение из `localStorage` не становится серверной игровой сессией на этапе 3. В базу переносится канонический общий инвентарь и проверенная модель эффектов; многопользовательские сессии появляются на этапе 7.

### 8. Исследовать и проверить Yandex Cloud

Основной вариант прототипа:

- [Yandex Container Registry](https://yandex.cloud/en/docs/container-registry/) для Docker-образов;
- [Yandex Serverless Containers](https://yandex.cloud/en/docs/serverless-containers/) для API;
- [Yandex Managed Service for PostgreSQL](https://yandex.cloud/en/docs/managed-postgresql/) для базы;
- [Yandex Lockbox](https://yandex.cloud/en/docs/lockbox/) для серверных секретов;
- [Yandex Object Storage](https://yandex.cloud/en/docs/storage/) для production-сборки фронтенда;
- API Gateway как единая точка входа для статического фронтенда и `/api/*`.

API Gateway поддерживает обе необходимые интеграции:

- [раздачу файлов из Object Storage](https://yandex.cloud/en/docs/api-gateway/concepts/extensions/object-storage);
- [вызов Serverless Containers](https://yandex.cloud/en/docs/api-gateway/concepts/extensions/containers).

Yandex документирует прямое подключение Serverless Containers к Managed PostgreSQL: [официальный сценарий](https://yandex.cloud/en/docs/tutorials/container-infrastructure/container-pg-connect).

Исследовательский прототип должен проверить:

- стоимость минимальной конфигурации;
- запуск Go-контейнера;
- подключение к PostgreSQL внутри облачной сети;
- передачу секретов из Lockbox;
- публичный HTTPS endpoint;
- раздачу React-сборки из Object Storage через API Gateway;
- маршрутизацию `/api/*` в Serverless Container;
- работу фронтенда и API с одного origin;
- fallback на `index.html` для обычных browser-маршрутов;
- применение миграций;
- smoke test после деплоя;
- допустимое время холодного старта.

### 9. Подготовить GitHub Actions

Минимальный набор workflow:

```text
frontend-ci.yml
backend-ci.yml
deploy-frontend.yml
deploy-backend.yml
```

`frontend-ci.yml`:

- работает из `frontend/`;
- запускает typecheck, тесты и production-сборку;
- не выполняет deployment.

`backend-ci.yml`:

- запускает Go-тесты;
- проверяет форматирование и статический анализ;
- поднимает тестовый PostgreSQL;
- применяет миграции;
- запускает интеграционные тесты;
- собирает Go-бинарник и Docker-образ.

`deploy-backend.yml`:

- запускается после успешных проверок основной ветки;
- собирает Docker-образ;
- использует уникальный тег commit SHA;
- публикует образ в Container Registry;
- применяет миграции;
- создаёт новую ревизию Serverless Container;
- выполняет smoke test.

`deploy-frontend.yml`:

- запускается после успешных frontend-проверок основной ветки;
- собирает production-фронтенд;
- загружает сборку в Object Storage;
- очищает или обновляет устаревшие файлы безопасным способом;
- выполняет smoke test через адрес API Gateway.

Для доступа GitHub Actions к Yandex Cloud исследуется [Workload Identity Federation](https://yandex.cloud/en/docs/iam/concepts/workload-identity), чтобы не хранить долгоживущий ключ сервисного аккаунта. Точная поддержка GitHub Actions подтверждается прототипом до фиксации workflow.

### 10. Развернуть и проверить бэкенд

- создать тестовое облачное окружение;
- применить миграции;
- импортировать исходный снимок;
- проверить API вручную и автоматически;
- провести smoke test публичного endpoint;
- зафиксировать адрес API для фронтенда;
- не переключать production-фронтенд до успешной приёмки.

### 11. Переключить фронтенд

После успешной приёмки сервиса:

- добавить типизированный API-клиент;
- заменить локальные импорты JSON;
- добавить состояния загрузки, ошибки и отсутствия данных;
- использовать `/api/*` с того же origin в production;
- проверить маршруты карты, героев, рас и регионов;
- сравнить интерфейс до и после переключения;
- подготовить bucket production-фронтенда;
- настроить API Gateway для статики и API;
- развернуть и проверить кандидата на production по техническому HTTPS-адресу;
- переключить production с GitHub Pages на Yandex Cloud;
- перестать публиковать GitHub Pages как рабочую версию сервиса;
- обновить документацию источников истины.

### 12. Завершить миграцию источника истины

- фронтенд больше не читает JSON;
- бэкенд не читает JSON во время обычной работы;
- PostgreSQL объявляется рабочим источником истины;
- `concepts/content-snapshot/` объявляется неизменяемым историческим снимком;
- новые изменения до появления админки выполняются миграциями данных;
- `AGENTS.md`, README и скиллы обновляются под новый порядок работы.

## Работа с визуальными концептами

Утверждённые исходные визуалы хранятся в `concepts/assets/` и индексируются manifest.

Для production-фронтенда используется контролируемый build-шаг:

```text
concepts/assets/
      ↓ проверка manifest
      ↓ оптимизация или копирование утверждённых файлов
frontend/public/assets/
```

Производные runtime-файлы не становятся новым каноном. В будущем их доставка может быть перенесена в Yandex Object Storage, но это не входит в этап 3.

## Тестирование

### Модульные тесты

- валидация моделей;
- преобразование доменных моделей в публичные DTO;
- обработка ошибок;
- правила публичности полей.

### Интеграционные тесты

- применение миграций на чистом PostgreSQL;
- импорт снимка;
- ограничения внешних ключей;
- HTTP API;
- OpenAPI-контракт;
- отсутствие мастерских полей.

### Проверка миграции

- автоматическое сравнение JSON-снимка и базы;
- автоматическое сравнение ожидаемых публичных данных и ответов API;
- отчёт по количеству и идентификаторам сущностей.

### Проверка фронтенда после переключения

- отсутствие ошибок в консоли;
- корректные loading, empty и error состояния;
- карта на высоте окна;
- маршруты карты, героев, рас и регионов;
- клики по регионам;
- одинаковый пользовательский контент до и после миграции;
- одинаковое поведение локального игрового цикла после перехода на API;
- сохранность общего инвентаря и машинно-читаемых эффектов;
- ширины 1024, 1280 px и мобильная компоновка;
- production-сборка загружается в Object Storage;
- маршруты открываются через API Gateway;
- запросы `/api/*` уходят с того же origin;
- GitHub Pages больше не является production-площадкой.

## Критерии завершения

- репозиторий разделён на `frontend/`, `backend/` и `concepts/`;
- существующий фронтенд работает после перемещения;
- чистая база разворачивается миграциями;
- весь JSON-снимок импортируется без потери фактов;
- все стабильные `id` сохранены;
- все ссылки защищены и валидны;
- публичное API не отдаёт секреты мастера;
- OpenAPI соответствует фактическим ответам;
- Go-тесты и интеграционные тесты проходят;
- Docker-образ собирается;
- бэкенд развёрнут в Yandex Cloud;
- GitHub Actions проверяет фронтенд и бэкенд;
- GitHub Actions публикует фронтенд и бэкенд в Yandex Cloud;
- production-фронтенд и API доступны через единый HTTPS-адрес;
- обычные browser-маршруты и их прямое открытие работают через API Gateway;
- GitHub Pages выведен из production;
- фронтенд получает все структурированные данные через API;
- локальный игровой цикл продолжает работать с данными API;
- общий инвентарь и игровые эффекты перенесены без потерь;
- внешний вид и маршруты не ухудшились после переключения;
- JSON не используется во время работы приложений;
- PostgreSQL зафиксирован как рабочий источник истины;
- README, `AGENTS.md` и локальные скиллы отражают новую архитектуру.

## Что не входит в этап

- создание и редактирование персонажей через интерфейс;
- загрузка пользовательских портретов;
- Яндекс ID;
- роли пользователей;
- интерфейс приглашений;
- редактирование кампаний через приложение;
- игровые комнаты;
- подключение игроков;
- сохранение состояния игровой сессии;
- отдельное хранение runtime-артов вне production-сборки фронтенда;
- резервное копирование.

## Риски и проверки

### Потеря или изменение канона

Снижается транзакционным импортом, сохранением `id`, автоматическим сравнением и неизменяемым JSON-снимком.

### Утечка секретов мастера

Снижается отдельными публичными DTO и тестами ответов API.

### Поломка фронтенда при переключении

Снижается предварительным деплоем API, типизированным клиентом и визуальным сравнением до/после.

### Стоимость Yandex Cloud

Проверяется на прототипе до создания постоянной инфраструктуры.

### Холодный старт Serverless Containers

Измеряется на прототипе. При неприемлемой задержке вариант размещения пересматривается до production-переключения.

### Рассинхронизация концептов и runtime-ассетов

Снижается manifest и детерминированным build-шагом подготовки ассетов.

## Открытые технические решения внутри этапа

Эти вопросы решаются отдельными техническими решениями и не меняют согласованную продуктовую границу:

- точная схема таблиц;
- перечень JSONB-полей;
- библиотека миграций;
- генератор OpenAPI-кода;
- подключение собственного домена вместо технического адреса API Gateway;
- минимальная конфигурация Managed PostgreSQL;
- точная спецификация маршрутов API Gateway;
- точный механизм GitHub Actions → Yandex Cloud;
- формат manifest для подготовки runtime-ассетов.

Любое решение, меняющее публичную модель данных, источник истины, стоимость инфраструктуры или границу секретов мастера, согласуется с владельцем проекта отдельно.
