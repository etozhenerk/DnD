# Инструкция для агентов

## Цель проекта

Развивать кликабельный сервис авторских D&D-кампаний для одного общего экрана. Мастер управляет страницей и рассказывает историю; авторизация, комнаты и отдельный режим мастера пока вне scope.

## Источники истины

1. `content/**/*.json` — канонические структурированные данные для приложения.
2. `content/campaigns/*-guide.md` — человекочитаемый сценарий мастера; он обязан соответствовать JSON.
3. `content/README.md` — схема и термины.
4. `assets/concepts/manifest.json` и `assets/concepts/` — утверждённые визуальные сущности и референсы.
5. `source-materials/` — приватный архив исходников, не канон и не часть Git; стандартные скиллы не должны от него зависеть.

При конфликте между исходным PDF и уже согласованным `content/` не исправлять данные молча. Описать конфликт пользователю и запросить решение.

## Обязательная маршрутизация по скиллам

- Любое изменение структуры, связей, правил или нескольких типов сущностей: прочитать и применить `skills/maintain-dnd-world/SKILL.md`.
- Новый герой или существенная переработка героя: `skills/create-dnd-character/SKILL.md`.
- Новый враг, NPC-противник, злодей, босс или боевая группа: `skills/create-dnd-enemy/SKILL.md`.
- Новая область карты, кампания, сюжетная ветка или набор локаций: `skills/create-dnd-campaign/SKILL.md`.
- Если задача пересекает области, применять `maintain-dnd-world` последним как проверку целостности.

Всегда сообщать пользователю, какой локальный скилл применяется. Не переносить ответы из одной кампании в другую без проверки.

## Правила контента

- Сохранять домашнюю облегчённую d20-систему из `content/rules.json`; не заменять её D&D 5e.
- Сохранять русский язык, личный юмор, абсурд и существующие местоимения персонажей.
- Если герой основан на реальном человеке, использовать его фотографию как обязательный визуальный референс: сохранять узнаваемость, согласовывать степень расовой трансформации и выдерживать стиль существующей команды и выбранной расы. Применять `skills/create-dnd-character/SKILL.md`.
- Использовать стабильные `id` в lower-kebab-case; не переименовывать существующий `id` без миграции всех ссылок.
- Разделять текст для мастера (`readAloud`, `description`) и машинный эффект (`check`, `effect`, `uses`).
- Для каждого ограниченного умения указывать область восстановления: `turn`, `round`, `battle`, `location` или `campaign`.
- Любой новый факт, влияющий на канон, отмечать в результате работы.
- Не добавлять секреты мастера в публично показываемый игрокам текст без явного поля/режима сокрытия.
- Не коммитить содержимое `source-materials/`, личные изображения, `.env`, зависимости или сборки.
- Коммитить можно только утверждённые производные концепты из `assets/concepts/`; исходные фотографии реальных людей там запрещены.
- Не считать сгенерированный портрет каноническим до явного одобрения пользователя.

## Обязательная проверка после изменений

1. Проверить синтаксис всех изменённых JSON через `jq empty`.
2. Проверить уникальность `id` и существование ссылок на героев, врагов, NPC, встречи и локации.
3. Сверить JSON кампании и соответствующий `*-guide.md`.
4. Проверить, что числа подходят группе из пяти героев либо явно указать другой размер группы.
5. Показать пользователю созданные сущности, принятые допущения и открытые вопросы.

## Изменения фронтенда

Когда появится приложение, оно должно читать данные из `content/`, а не содержать копии лора внутри компонентов. Новые поля сначала описывать в `content/README.md`, затем добавлять в данные и интерфейс.
