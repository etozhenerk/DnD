# Этап 5. Авторизация и владение персонажами

Статус: согласован.

Короткое название: Вход для друзей.

Краткое описание: Подключаем Яндекс ID и закрепляем персонажей за их владельцами.

Родительский документ: [Продуктовый roadmap](../product-roadmap.md).

## Цель

Заменить общий пароль редактора персональным входом через Яндекс ID, закрыть сервис для посторонних и дать друзьям возможность создавать и редактировать принадлежащих им персонажей.

На этом этапе появляются пользователи и права на персонажей, но ещё не появляются игровые комнаты, партии и роли мастера или игрока внутри конкретной игры.

## Результат для пользователя

Участник компании может:

- войти через свой Яндекс ID;
- получить доступ ко всему содержимому закрытого сервиса;
- увидеть принадлежащих ему персонажей;
- создать нового персонажа;
- продолжить свой черновик;
- подготовить новую ревизию своего опубликованного персонажа;
- выйти из сервиса.

Администратор дополнительно может:

- видеть и редактировать всех персонажей;
- назначать и менять владельца персонажа;
- публиковать подготовленные ревизии;
- блокировать доступ пользователя через ручную административную операцию.

## Согласованные решения

- Для входа используется Яндекс ID.
- Сервис доступен только заранее разрешённым друзьям.
- Интерфейса приглашений и самостоятельной регистрации нет.
- Список разрешённых аккаунтов изменяется вручную через бэкенд.
- После появления авторизации весь контент закрывается серверной сессией.
- Без авторизации доступны только экран входа и технические OAuth-маршруты.
- Глобальные роли сервиса: `admin` и `member`.
- Роли `master` и `player` будут контекстными и появятся вместе с игровыми сессиями.
- У персонажа может быть один владелец или не быть владельца.
- Администратор может назначать и переназначать владельца.
- Участник создаёт персонажей и редактирует только принадлежащих ему.
- Администратор сохраняет право редактировать и публиковать всех персонажей.
- Общий пароль редактора удаляется после проверки первого аккаунта администратора.
- Яндекс OAuth-токены не хранятся в браузере и не используются как сессия приложения.

## 1. Пользовательский сценарий входа

```text
Экран входа
    ↓
«Войти через Яндекс»
    ↓
Яндекс ID: подтверждение входа
    ↓
Callback бэкенда
    ↓
Получение стабильного Yandex user ID
    ↓
Проверка ручного allowlist
    ├── доступ разрешён → локальный пользователь и сессия → приложение
    └── доступа нет      → нейтральный экран отказа
```

Используется серверный Authorization Code flow с `state` и PKCE. Яндекс ID поддерживает PKCE и рекомендует не хранить OAuth-токен в браузере или открытой конфигурации: [получение кода](https://yandex.ru/dev/id/doc/en/codes/code-url).

Бэкенд использует OAuth-токен только для запроса данных пользователя, получает стабильный `id` и затем создаёт собственную серверную сессию: [данные пользователя](https://yandex.ru/dev/id/doc/en/user-information).

### Минимальные данные профиля

Сервис хранит только:

- стабильный Yandex user ID;
- актуальный логин;
- отображаемое имя;
- локальную роль;
- статус доступа;
- даты первого и последнего входа.

Телефон, дата рождения и другие необязательные данные не запрашиваются. Аватар Яндекс ID в первую версию не входит.

## 2. Ручной список доступа

Для доступа используется отдельный allowlist в PostgreSQL.

Предлагаемые административные команды:

```text
backend access allow <yandex-login> --role member
backend access allow <yandex-login> --role admin
backend access block <yandex-login>
backend access list
```

Команды выполняются человеком с административным доступом к инфраструктуре. Пользовательского интерфейса управления приглашениями нет.

### Первое связывание

1. Администратор заранее добавляет логин друга в allowlist.
2. Друг входит через Яндекс ID.
3. Бэкенд сравнивает полученный логин с незакреплённой записью.
4. Запись навсегда связывается со стабильным Yandex user ID.
5. Последующие входы проверяются по ID, а не только по логину.

Если аккаунт заблокирован или отсутствует в allowlist, API не отдаёт содержимое мира. Экран отказа не содержит формы запроса приглашения и внутренних причин блокировки.

## 3. Сессия приложения

После успешного входа бэкенд создаёт собственную случайную сессию:

- идентификатор сессии передаётся только в `HttpOnly Secure` cookie;
- cookie использует подходящий `SameSite`-режим;
- сервер хранит только безопасное представление идентификатора сессии;
- сессия имеет ограниченный срок жизни;
- выход отзывает текущую сессию;
- блокировка пользователя отзывает все его сессии;
- write-запросы защищаются от CSRF с учётом выбранного cookie-режима.

Yandex access token не передаётся фронтенду и удаляется после получения данных профиля. Refresh token не запрашивается, если повторное обращение к API Яндекс ID не требуется.

Технический HTTPS-адрес и единый origin фронтенда и `/api/*` уже подготовлены на этапе 3.

## 4. Роли и права

### `member`

Может:

- читать содержимое закрытого сервиса;
- создавать собственного персонажа;
- редактировать свой черновик;
- создавать новую ревизию своего опубликованного персонажа;
- загружать готовый портрет для своего персонажа;
- просматривать историю своих ревизий.

Не может:

- редактировать чужого персонажа;
- менять владельца;
- публиковать ревизию в канон без администратора;
- менять allowlist или роли;
- получать мастерские секреты кампаний только на основании роли `member`.

### `admin`

Может выполнять все операции с персонажами, доступные в редакторе этапа 4, а также:

- назначать персонажа пользователю;
- менять владельца;
- оставлять персонажа без владельца;
- просматривать и публиковать пользовательские ревизии.

Изменение allowlist остаётся серверной административной операцией и не переносится в UI даже для `admin`.

## 5. Владение персонажами

У `characters` появляется необязательное поле владельца.

Правила:

- новый персонаж участника сразу получает этого участника владельцем;
- существующие персонажи после миграции остаются без владельца;
- администратор вручную распределяет существующих героев;
- у персонажа одновременно только один владелец;
- смена владельца не создаёт новую каноническую ревизию, потому что не меняет лор и характеристики;
- смена владельца фиксируется в журнале административных действий;
- архивный персонаж сохраняет владельца;
- удаление пользователя не удаляет персонажа.

Если доступ пользователя заблокирован, его персонажи сохраняются. Администратор может оставить их закреплёнными, снять владельца или назначить другого человека.

## 6. Изменения интерфейса

### Публично доступные маршруты

```text
/login
/access-denied
```

OAuth callback обрабатывается отдельным серверным HTTP-маршрутом.

### Авторизованная часть

- в верхнем меню появляется компактный профиль и выход;
- неавторизованный пользователь перенаправляется на экран входа;
- книга героев может отмечать персонажей текущего пользователя;
- редактор показывает вкладки «Мои персонажи» и, для администратора, «Все персонажи»;
- запрещённые действия не только скрываются в UI, но и отклоняются API;
- состояние истёкшей сессии возвращает пользователя на вход без потери уже сохранённого черновика.

Экран входа остаётся простым и не раскрывает лор закрытого сервиса.

## 7. Предлагаемая модель данных

```text
allowed_accounts
users
user_sessions
characters.owner_user_id
access_audit_log
```

### `allowed_accounts`

Предварительно разрешённый логин, связанный Yandex user ID, назначаемая роль и статус доступа.

### `users`

Локальный пользователь сервиса со стабильной внутренней идентичностью. Yandex user ID уникален.

### `user_sessions`

Отзываемые серверные сессии с владельцем, сроком жизни и датой последнего использования.

### `characters.owner_user_id`

Необязательная ссылка на единственного владельца персонажа.

### `access_audit_log`

Журнал связывания аккаунта, изменения роли, блокировки и переназначения персонажа. OAuth-токены и cookie в журнал не попадают.

Точные имена таблиц и полей фиксируются техническим решением и миграциями.

## 8. API

Предлагаемые маршруты авторизации:

```text
GET    /api/v1/auth/yandex/start
GET    /api/v1/auth/yandex/callback
GET    /api/v1/auth/me
POST   /api/v1/auth/logout
```

Предлагаемые операции владения:

```text
GET    /api/v1/me/characters
POST   /api/v1/me/characters
PATCH  /api/v1/me/characters/{id}/draft
GET    /api/v1/admin/characters
PUT    /api/v1/admin/characters/{id}/owner
DELETE /api/v1/admin/characters/{id}/owner
POST   /api/v1/admin/characters/{id}/publish
```

Это ориентиры ресурсов, а не окончательный OpenAPI-контракт. Перед реализацией необходимо убрать дублирование с editor endpoints этапа 4 и выбрать единый последовательный API.

Все содержательные endpoints после завершения этапа требуют действующую сессию. Health endpoints могут оставаться технически доступными без пользовательской авторизации и не возвращают данные мира.

## 9. Переход с общего пароля

Чтобы не потерять административный доступ:

1. Добавить таблицы пользователей, allowlist и новые endpoints.
2. Внести первый аккаунт с ролью `admin`.
3. Выполнить полный вход через Яндекс ID в production.
4. Проверить чтение, редактирование и публикацию персонажа.
5. Проверить административное назначение владельца.
6. Только после этого отключить общий пароль редактора.
7. Удалить старый endpoint и серверную конфигурацию пароля.
8. Отозвать все старые `editor_sessions`.

Постоянного параллельного существования двух способов входа не будет.

## 10. Порядок реализации

1. Зарегистрировать OAuth-приложение Яндекс ID и callback URL.
2. Добавить серверную конфигурацию OAuth через Lockbox.
3. Создать миграции `allowed_accounts`, `users`, `user_sessions` и журнала.
4. Реализовать Authorization Code flow, PKCE и проверку `state`.
5. Реализовать получение минимального профиля и связывание аккаунта.
6. Реализовать локальные сессии, выход и отзыв.
7. Закрыть содержательные API endpoints авторизацией.
8. Реализовать `admin`/`member` middleware и проверки доступа к объекту.
9. Добавить владельца персонажа и миграцию существующих данных.
10. Адаптировать редактор под «Мои» и «Все персонажи».
11. Добавить экран входа, отказа и профиль пользователя.
12. Создать и проверить первый аккаунт администратора.
13. Отключить общий пароль редактора.
14. Провести серверные, интеграционные и браузерные проверки.

## Тестирование

### Авторизация

- успешный вход разрешённого пользователя;
- отказ пользователю вне allowlist;
- связывание логина со стабильным ID;
- повторный вход после изменения отображаемого имени;
- проверка `state` и PKCE;
- истечение и отзыв сессии;
- выход;
- блокировка пользователя;
- отсутствие OAuth-токена в браузере, логах и базе;
- невозможность получить контент без сессии.

### Права

- участник создаёт собственного персонажа;
- участник редактирует своего персонажа;
- участник не может изменить чужого персонажа прямым API-запросом;
- участник не может самостоятельно публиковать ревизию;
- администратор редактирует и публикует любого персонажа;
- администратор назначает и меняет владельца;
- смена владельца не меняет каноническую ревизию;
- блокировка владельца не удаляет персонажа.

### Фронтенд

- редирект на вход без сессии;
- успешный возврат после Яндекс ID;
- нейтральный отказ в доступе;
- корректное истечение сессии;
- профиль и выход;
- списки «Мои персонажи» и «Все персонажи»;
- отсутствие административных действий у `member`;
- сохранение уже записанного черновика после повторного входа;
- отсутствие ошибок в консоли и горизонтального переполнения.

## Критерии завершения

- весь пользовательский контент закрыт серверной авторизацией;
- разрешённый друг входит через Яндекс ID;
- пользователь вне allowlist не получает данные мира;
- существует хотя бы один проверенный аккаунт `admin`;
- общий пароль и старые editor-сессии удалены;
- роли проверяются бэкендом;
- участник создаёт и редактирует собственного персонажа;
- участник не может изменить чужого персонажа;
- администратор управляет всеми персонажами и владельцами;
- существующие персонажи корректно мигрированы без владельца;
- OAuth-токены не хранятся в браузере и долговременно в базе;
- OpenAPI, README и эксплуатационная инструкция обновлены;
- серверные, интеграционные и браузерные тесты проходят.

## Что не входит в этап

- публичная регистрация;
- интерфейс приглашений;
- восстановление отдельного пароля сервиса;
- вход через другие OAuth-провайдеры;
- роль мастера или игрока на уровне всего сервиса;
- выбор персонажа для партии;
- добавление персонажа в кампанию;
- игровые комнаты;
- дистанционное подключение игроков;
- синхронизация активной сцены;
- редактор кампаний;
- AI-функции.

## Открытые технические решения внутри этапа

- точное время жизни сессии;
- правила продления активной сессии;
- способ хранения хеша идентификатора сессии;
- конкретная CSRF-защита;
- формат административной CLI-команды;
- срок хранения журнала доступа;
- политика конкурентного редактирования персонажа владельцем и администратором.

Эти решения не меняют согласованную продуктовую модель. Любое расширение собираемых данных Яндекс ID, появление второго владельца персонажа или изменение права публикации требует отдельного согласования.
